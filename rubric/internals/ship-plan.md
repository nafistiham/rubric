## Phase 1: Todo Baseline Mechanism

**Goal:** Let users adopt rubric on existing codebases without fixing every violation first.

**Why this is first:** A project like sidekiq has 1329 genuine violations. No team will run a linter in CI that starts with 1329 red flags. RuboCop solved this with `.rubocop_todo.yml`. We need the same.

### Tasks

**Task 1.1** — Add `rubric todo` subcommand
- Location: `rubric-cli/src/main.rs`
- Behavior: run all enabled cops (same as `rubric check`), collect all diagnostics, write `.rubric_todo.toml`
- Format:
  ```toml
  # Auto-generated by `rubric todo`. Commit this file.
  # Run `rubric todo --regenerate` after fixing violations.

  [todo."Style/Documentation"]
  files = ["lib/sidekiq/api.rb", "lib/sidekiq/job.rb"]

  [todo."Layout/SpaceInsideHashLiteralBraces"]
  files = ["lib/sidekiq/api.rb", "lib/sidekiq/client.rb"]
  ```
- Each cop section lists the files that have violations for that cop
- Wire up via `match args[0] { "todo" => run_todo(...) }`

**Task 1.2** — Suppress todo violations in `rubric check`
- On startup, look for `.rubric_todo.toml` in the target directory (same logic as `rubric.toml` discovery)
- For each diagnostic: if `(cop, file)` is in the todo map → suppress it
- A file with NEW violations (not in todo) still shows them
- A file with violations reduced below the todo count: still suppressed (conservative — don't punish cleanup)

**Task 1.3** — Add `--regenerate-todo` flag to `rubric check`
- `rubric check --regenerate-todo` re-runs and rewrites `.rubric_todo.toml` with current state
- Replaces the old file entirely
- Print: `Updated .rubric_todo.toml (added N, removed M entries)`

**Task 1.4** — Add `--ignore-todo` flag
- `rubric check --ignore-todo` runs without loading `.rubric_todo.toml`
- Shows all violations including suppressed ones
- Useful for: seeing total debt, auditing, CI jobs that care about the full picture

**Task 1.5** — Write `docs/todo-baseline.md`
- Covers: why this exists, the workflow (migrate → todo → check → fix → regenerate), what to commit
- Add a one-liner in README pointing to it

### Tests

Write tests in `rubric-cli/tests/todo_test.rs`:

```
test_todo_generates_file:
  - Create temp dir with 2 Ruby files, each with 1 Style/StringLiterals violation
  - Run rubric todo on it
  - Assert .rubric_todo.toml exists and contains both files under Style/StringLiterals

test_todo_suppresses_known_violations:
  - Same setup as above
  - Run rubric check → should report 0 violations (both suppressed)

test_todo_surfaces_new_violations:
  - Generate .rubric_todo.toml for file A (1 violation)
  - Add a NEW violation type to file A (e.g. Style/Documentation)
  - Run rubric check → should report 1 violation (the new one), suppress the old

test_ignore_todo_flag:
  - Same setup, with .rubric_todo.toml present
  - Run rubric check --ignore-todo → reports all violations

test_regenerate_flag:
  - Generate .rubric_todo.toml
  - Fix one violation in a file
  - Run rubric check --regenerate-todo
  - Assert .rubric_todo.toml no longer contains that file for that cop
```

### Definition of Done

- [x] `rubric todo` generates `.rubric_todo.toml`
- [x] `rubric check` with todo file present reports 0 for sidekiq (1329 violations suppressed)
- [x] Adding a new violation to a sidekiq file → check reports 1
- [x] `--regenerate-todo` and `--ignore-todo` flags work
- [x] All 5 tests pass

## Phase 2: Distribution Fix

**Goal:** `gem install rubric-ruby && rubric check .` works on macOS and Linux with no Rust installed.

**Why this matters:** Right now, using rubric requires cloning the repo and running cargo build. That is a complete non-starter for any Ruby developer who doesn't write Rust.

### Tasks

**Task 2.1** — Cross-compilation setup
- Add `cross` to the build toolchain
- Define 4 targets: `x86_64-unknown-linux-gnu`, `aarch64-unknown-linux-gnu`, `x86_64-apple-darwin`, `aarch64-apple-darwin`
- Verify all 4 build locally with: `cross build --release --target <target> -p rubric-cli`

**Task 2.2** — GitHub Actions release workflow
- File: `.github/workflows/release.yml`
- Trigger: push of tag `v*`
- Steps: checkout, install cross, build all 4 targets, create GitHub Release, upload `rubric-<version>-<target>.tar.gz` as release assets
- Each tar contains: `rubric` binary + `LICENSE`

**Task 2.3** — Thin Ruby gem (`rubric-ruby`)
- Location: `gem/` directory (already exists, inspect first)
- `lib/rubric.rb`: detect platform (`RUBY_PLATFORM`), map to target name, check if binary exists in `bin/`, if not → download correct release asset from GitHub, extract, chmod +x
- `bin/rubric`: shim that exec's the downloaded binary
- `rubric-ruby.gemspec`: no runtime dependencies (download happens lazily)
- Version: keep in sync with Cargo.toml version

**Task 2.4** — Version sync script
- `scripts/bump-version.sh <new-version>` updates Cargo.toml + gemspec + CHANGELOG in one shot

**Task 2.5** — Test on clean machine
- Use Docker: `docker run --rm -it ruby:3.3` → `gem install rubric-ruby` → `rubric check /some/ruby/file.rb`
- Must work. No Rust. No cargo. No cloning.

### Tests

```
test_gem_install_downloads_binary:
  - In Docker (ruby:3.3, no Rust)
  - gem install rubric-ruby
  - which rubric → should resolve
  - rubric --version → should print version

test_gem_install_linux_aarch64:
  - Same test on linux/arm64

test_check_works_after_gem_install:
  - rubric check on a Ruby file with a known violation → reports it correctly
```

### Definition of Done

- [ ] All 4 cross-compile targets build successfully in CI
- [ ] GitHub release has all 4 binary assets
- [ ] `gem install rubric-ruby` on fresh Ruby Docker image (no Rust) works
- [ ] `rubric check` on a Ruby file reports violations correctly after gem install
- [ ] README install section updated to `gem install rubric-ruby`

**Estimated effort: 4 days**

## Phase 3: LSP Server

**Goal:** Violations appear as squiggles in VS Code within 100ms of typing. This is where the speed advantage is felt directly.

**Why this matters:** Developers experience linting in their editor, not in CI. RuboCop's editor integration is slow enough that many teams run it only on save or not at all. Rubric at 8-13x faster can be genuinely real-time.

### Tasks

**Task 3.1** — Create `rubric-lsp` crate
- `Cargo.toml` workspace member: `rubric-lsp`
- Dependencies: `tower-lsp`, `tokio`, `serde_json`
- Re-export the rubric-rules engine — no duplication

**Task 3.2** — Implement core LSP handlers
- `initialize`: return server capabilities (diagnostics only, no code actions in v1)
- `textDocument/didOpen`: parse file, run cops, publish diagnostics
- `textDocument/didChange`: re-run on each change (use incremental content from params)
- `textDocument/didSave`: re-run (catches config changes that might affect results)
- `textDocument/didClose`: clear diagnostics

**Task 3.3** — Wire up config and todo suppression
- On `initialize`, locate `rubric.toml` by walking up from `rootUri`
- Load `.rubric_todo.toml` if present
- Apply suppression to diagnostics before publishing
- Watch for `rubric.toml` changes via `workspace/didChangeWatchedFiles` — reload config

**Task 3.4** — Add `rubric lsp` subcommand
- In `rubric-cli/src/main.rs`: `"lsp" => run_lsp_server()`
- Server runs on stdio (language clients connect via stdio, not TCP)
- Log to stderr only (stdout is the JSON-RPC channel)

**Task 3.5** — VS Code extension
- Directory: `editors/vscode/`
- `package.json`: minimal extension, activates on `*.rb` files
- `extension.ts`: launch `rubric lsp` as the language server
- Install: `vsce package` → `.vsix` file distributed via GitHub releases alongside binaries
- **Do not publish to VS Code Marketplace in v1** — too much maintenance surface

**Task 3.6** — Editor setup documentation
- `docs/editor-setup.md`
- Cover: VS Code (extension), Neovim (nvim-lspconfig entry), Emacs (lsp-mode config)
- Each section: 3-5 lines. Not a tutorial — just the config snippet.

### Tests

```
test_lsp_publishes_diagnostics_on_open:
  - Start LSP server (stdio, child process)
  - Send initialize + initialized + textDocument/didOpen with Ruby content containing 1 violation
  - Assert textDocument/publishDiagnostics received with 1 diagnostic
  - Assert diagnostic range matches expected line/col

test_lsp_clears_on_fix:
  - Same setup, then send didChange with the violation fixed
  - Assert publishDiagnostics with 0 diagnostics

test_lsp_respects_rubric_toml:
  - Open a file with Style/Documentation violation
  - rubric.toml in rootUri has Style/Documentation disabled
  - Assert 0 diagnostics published

test_lsp_respects_todo_suppression:
  - .rubric_todo.toml suppresses the file's cop
  - Assert 0 diagnostics published

test_lsp_latency:
  - Send didChange, measure time until publishDiagnostics received
  - Assert < 200ms for a 200-line Ruby file
```

### Definition of Done

- [ ] `rubric lsp` starts and speaks LSP over stdio
- [ ] VS Code extension installed → squiggles appear in `.rb` files within 200ms
- [ ] Todo-suppressed violations do not appear as squiggles
- [ ] Neovim config documented and tested manually
- [ ] All 5 tests pass

**Estimated effort: 8 days**

## Phase 4: Real Users (3–5 Projects)

**Goal:** 3 open-source Ruby projects have rubric running in CI, with maintainer consent.

**Why this matters:** 42 sessions of solo development has no feedback loop. Real codebases will reveal bugs that the benchmark suite cannot. Without real users, the blog post is noise.

### Process

**Task 4.1** — Identify 5 candidate projects
Criteria:
- 200–2000 Ruby files
- Active maintainers (commit in last 3 months)
- Uses RuboCop currently
- Existing `.rubocop.yml` (for migrate to work)
- Open to tooling experiments (check CONTRIBUTING.md)

Do NOT target: Rails itself, Discourse, large orgs with slow PR processes.

Good targets: mid-size gems, dev tools, CLI utilities.

**Task 4.2** — For each candidate, prepare a PR
Steps:
1. Fork the repo
2. `rubric migrate` → produces `rubric.toml`
3. `rubric todo` → produces `.rubric_todo.toml`
4. Add to CI (GitHub Actions example):
   ```yaml
   - name: Lint with Rubric
     run: |
       gem install rubric-ruby
       rubric check .
   ```
5. Open PR with title: "Add rubric linter (fast Ruby linter written in Rust)"
6. PR description: what rubric is, why we're trying it, what the todo file is for

**Task 4.3** — Fix bugs that surface
Every crash or incorrect diagnostic reported by maintainers is a P0. Fix before sending next PRs.
Track all reports in GitHub Issues with label `user-reported`.

**Task 4.4** — Collect structured feedback
After 1 week, ask maintainers:
- What confused you?
- Did the migration feel correct?
- Any violations that seemed wrong?
- Would you keep it in CI?

### Tests (validation criteria)

## Phase 5: `inherit_gem` Support in Migrate

**Goal:** `rubric migrate` correctly handles projects that use shared RuboCop configs via `inherit_gem`.

**Why this matters:** Most production Ruby projects in 2026 use shared configs via gems (rubocop-shopify, standard, rubocop-rails-omakase). The current migrate silently ignores these, producing a rubric.toml that enables cops the effective config disables. This is the most common migration failure mode.

### Tasks

**Task 5.1** — Parse `inherit_gem:` in YAML loader
- Location: `rubric-cli/src/migrate.rs`
- Parse `inherit_gem:` as a map of `gem_name → path_within_gem`
- Don't error — this is now handled instead of warned

**Task 5.2** — Resolve gem paths
- Shell out: `ruby -e "puts Gem::Specification.find_by_name('GEM_NAME').gem_dir"` for each gem
- If Ruby not found or gem not installed → print warning, continue (graceful degradation)
- Construct full path: `gem_dir + "/" + path_within_gem`

**Task 5.3** — Merge gem rubocop.yml as base config
- Load the gem's `rubocop.yml` using the same YAML parser
- Apply it first, then overlay the project's own `.rubocop.yml` on top
- Handle multiple `inherit_gem` entries in order (same as RuboCop's inheritance)
- Handle recursive `inherit_from:` within the gem's config (shallow only — 1 level)

**Task 5.4** — Priority target: the 3 most common gems
- `rubocop-shopify` → disables ~200 cops including LineLength
- `standard` → enables specific cop set with different defaults
- `rubocop-rails-omakase` → enables Rails-specific cops
- Write integration tests for each

**Task 5.5** — Add to migration summary output
```
Migrated: .rubocop.yml → rubric.toml
  inherit_gem resolved: rubocop-shopify (147 cops affected)
  WARNING: inherit_gem resolved: rubocop-custom (gem not installed, skipped)
```

### Tests

```
test_inherit_gem_shopify:
  - .rubocop.yml: inherit_gem: { rubocop-shopify: rubocop.yml }
  - Run rubric migrate
  - Assert rubric.toml has Layout/LineLength disabled (shopify disables it)
  - Assert rubric.toml has Style/Documentation disabled

test_inherit_gem_multiple:
  - .rubocop.yml: inherit_gem with 2 gems, project also overrides 1 cop
  - Assert final rubric.toml reflects all 3 layers

test_inherit_gem_gem_not_installed:
  - .rubocop.yml: inherit_gem with a gem that isn't installed
  - Assert rubric migrate succeeds with a warning, not a crash

test_inherit_gem_then_project_override:
  - Gem disables CopX. Project re-enables CopX: Enabled: true
  - Assert rubric.toml has CopX enabled (project wins)
```

### Definition of Done

- [ ] `rubric migrate` on a rubocop-shopify project produces correct rubric.toml
- [ ] `rubric migrate` on a standard project produces correct rubric.toml
- [ ] All 4 tests pass
- [ ] Warning message shown for unresolved gems, no crash

**Estimated effort: 4 days**
